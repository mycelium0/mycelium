# `myceliumctl` ŌĆö Phase 0/1 control tooling

Author: mindicator & silicon bags quartet.

`myceliumctl` is a POSIX/bash + [`jq`](https://jqlang.github.io/jq/) command-line tool. It is the
rendering engine for a persistent private network node's data plane: it issues client identities,
generates REALITY parameters, renders the **server config** for one of two transport engines, and
emits per-client subscription configs for off-the-shelf clients (sing-box, Clash-Meta).

## Engines: sing-box (primary) ┬Ę xray (optional)

The tool drives **two** rendering engines, selected with `--engine`:

| Engine | Flag | Role | Protocols |
|---|---|---|---|
| **sing-box** | `--engine singbox` | **PRIMARY** ŌĆö one server, many toggleable protocols | VLESS+REALITY (Vision / gRPC / XHTTP), Hysteria2, TUIC v5, Shadowsocks-2022, ShadowTLS v3, Trojan |
| **xray** | `--engine xray` | **OPTIONAL** alternative | VLESS + XTLS-Vision + REALITY only |

`sing-box` is the **primary** engine and the **default**: a single sing-box process speaks every
**enabled** protocol, and each protocol is **individually toggleable** via a `<proto>_enabled` flag
in the params file, so an operator exposes only a chosen subset. `xray` remains available as an
optional alternative engine for the single-transport path (select it with `--engine xray`); the
dataplane `nodes/dataplane/vless-reality/` component is not removed.

> **Default engine.** The default engine is **`sing-box`** (project canon). Select the optional
> alternative with `--engine xray`. If you invoke `render-server`/`subscription` without `--engine`,
> the tool prints one INFO line noting that the default is now `singbox` (the historical default was
> `xray`). Pin engine versions: **sing-box Ōēź `v1.11.x`** (record the exact deployed tag at deploy
> time) and **xray Ōēź `v26.2.4`**.

There is **no Go** in this tool and **no custom cryptography**. Key material comes only from
audited, off-the-shelf tools:

- REALITY X25519 keypairs and client UUIDs come from **`xray`** (`xray x25519`, `xray uuid`) ŌĆö or
  the equivalent **`sing-box generate`** subcommands (`reality-keypair`, `uuid`).
- REALITY `shortIds` and protocol passwords come from **`openssl rand`** (or `sing-box generate
  rand`). Shadowsocks-2022 uses base64 PSKs of the method's key length.

The tool itself never invents key material ŌĆö `render-server`/`subscription` only place values
supplied in the params file (or generated by `reality-keys`) into the config **by `jq` path**.
See [`docs/adr/0002-no-custom-cryptography.md`](../docs/adr/0002-no-custom-cryptography.md).

The runtime **control-agent** ŌĆö network-state detection, auto-rotation of transport/port/SNI/keys,
telemetry and policy ŌĆö is intentionally **not** part of this tool. It is a **Phase-2 deliverable in
Go or Rust** (see [`docs/ARCHITECTURE.md`](../docs/ARCHITECTURE.md) Layer 2 and the Phase-0
proposal). `myceliumctl` covers only the manual, operator-driven Phase-0 surface.

## Requirements

| Tool | Needed for | Notes |
|---|---|---|
| `bash` | everything | POSIX-style; tested against bash 3.2 (macOS) and newer |
| `jq` | everything | the rendering engine; mandatory for every command |
| `xray` (Ōēź v26.2.4) | `reality-keys`, `identity add` | provides `xray x25519` and `xray uuid` |
| `openssl` | `reality-keys` | `openssl rand -hex` for shortIds / passwords |
| `yq` | params files written in YAML | optional; JSON params always work without it |

`render-server` and `subscription` need only `bash` + `jq` for **either** engine ŌĆö they place values
that already exist in the params file (no key generation at render time). `reality-keys` and
`identity add` need `xray` + `openssl`; if `xray` is missing they fail with a clear error and
generate **nothing** ŌĆö the tool never falls back to hand-rolled key generation. The deployed
**server** binaries are pinned separately: **sing-box Ōēź `v1.11.x`** (primary) and/or
**xray Ōēź `v26.2.4`** (optional).

## Commands

```
myceliumctl reality-keys [--shortids N] [--shortid-bytes B]
myceliumctl identity add    --name NAME [--state FILE]
myceliumctl identity revoke NAME|ID    [--state FILE]
myceliumctl identity list              [--state FILE]
myceliumctl render-server [--engine singbox|xray] [--template FILE] --params FILE [--state FILE] [--out FILE]
myceliumctl subscription  [--engine singbox|xray] --params FILE [--state FILE] [--out DIR]
myceliumctl help
```

`--engine` defaults to `singbox` (primary; project canon); pass `--engine xray` for the optional
alternative. The default `--template` is engine-specific:
`nodes/dataplane/singbox/server.template.json` for sing-box and
`nodes/dataplane/vless-reality/server.template.json` for xray. Pass `--template` to override.

### `reality-keys`

Generates a REALITY keypair and one or more shortIds, printed as `KEY=VALUE` lines:

```
$ myceliumctl reality-keys --shortids 2
REALITY_PRIVATE_KEY=...
REALITY_PUBLIC_KEY=...
REALITY_SHORT_ID_1=...
REALITY_SHORT_ID_2=...
```

`--shortid-bytes B` controls shortId length (default 8 bytes ŌåÆ 16 hex chars). Pipe the output into
your params file or a gitignored secrets file; never commit it.

### `identity`

Maintains a JSON state file of clients. Default state file:
`control/state/identities.json` (gitignored). Each record is
`{ "name": ..., "id": <xray uuid>, "created": <RFC3339 UTC> }`. All edits go through `jq`; a failed
filter can never corrupt the file (it is validated before replacing the original). Duplicate names
and duplicate UUIDs are rejected. `revoke` accepts either the name or the UUID.

### `render-server`

Reads a **server template**, a **params** file, and the **identity state**, and writes a concrete
`server.json` by editing JSON **by `jq` path** ŌĆö it never string-splices values into the config.
Default output: `control/state/server.json` (gitignored).

#### `--engine singbox` (primary, default)

Renders one sing-box server carrying every **enabled** protocol. Inbounds are located in the
template by their **`tag`** (robust to ordering), filled by `jq` path, and any inbound whose
protocol is not enabled in params is **pruned** from the output. Per matching inbound it sets:

- `tls.reality.private_key` ŌåÉ `params.reality_private_key`; `tls.reality.short_id[]` ŌåÉ
  `params.short_ids`; `tls.server_name` ŌåÉ `params.donor_sni`; `tls.reality.handshake.server` ŌåÉ
  `params.donor_host` (for the three VLESS+REALITY inbounds).
- `tls.server_name`/`tls.certificate_path`/`tls.key_path` ŌåÉ `params.tls_sni` / `tls_certificate_path`
  / `tls_key_path` (for the TLS-cert protocols Hysteria2 / TUIC / Trojan).
- `listen_port` ŌåÉ the per-protocol `<proto>_port`.
- `transport.service_name` ŌåÉ `params.grpc_service_name` (gRPC); `transport.path` ŌåÉ
  `params.xhttp_path` (XHTTP).
- `password` ŌåÉ `params.ss_password` (Shadowsocks-2022 and the inner SS of ShadowTLS);
  `handshake.server` ŌåÉ `params.shadowtls_handshake_server` (ShadowTLS).
- `users[]` ŌåÉ every client in state, in the per-protocol shape:
  VLESS ŌĆö `{ name, uuid, flow }` (`flow: "xtls-rprx-vision"` for Vision/TCP, empty for gRPC/XHTTP);
  TUIC ŌĆö `{ name, uuid, password }`; Hysteria2 / Trojan / ShadowTLS ŌĆö `{ name, password }`;
  Shadowsocks-2022 ŌĆö `{ name, password }` (multi-user). A per-identity `password` in state, if
  present, overrides the shared per-protocol secret.

Default template path: `../nodes/dataplane/singbox/server.template.json` (produced by the dataplane
component; `control/testdata/singbox.server.template.json` is a standalone fixture for the
self-test).

#### `--engine xray` (optional alternative)

Reads the dataplane VLESS + REALITY template and sets, on the first `vless` inbound:
`ŌĆ”realitySettings.privateKey` ŌåÉ `reality_private_key`; `ŌĆ”shortIds` ŌåÉ `short_ids`; `ŌĆ”serverNames` ŌåÉ
`[ donor_sni ]`; `ŌĆ”dest` ŌåÉ `dest` (default `donor_host:443`); `ŌĆ”port` ŌåÉ `listen_port`;
`ŌĆ”settings.clients` ŌåÉ every client, each with `flow: "xtls-rprx-vision"`. Default template path:
`../nodes/dataplane/vless-reality/server.template.json`.

### `subscription`

Emits, per client, two files into the output directory (default `control/out`, gitignored). Clients
receive only the REALITY **public** key ŌĆö never the private key.

#### `--engine singbox` (primary, default)

- `<name>.singbox.json` ŌĆö one **outbound per enabled protocol** (`vless` with Vision/gRPC/XHTTP,
  `hysteria2`, `tuic`, `shadowsocks`, `shadowtls`, `trojan`), plus a `urltest` outbound (`tag:
  "auto"`) and a `selector` outbound (`tag: "mycelium"`, `default: "auto"`) that prefer the
  protocols in **priority order** (REALITY/TCP first, UDP/QUIC next, SS/ShadowTLS, then Trojan).
  ShadowTLS is emitted as a routable Shadowsocks outbound with a `detour` to a hidden
  `shadowtls-handshake` outbound (the v3 handshake partner is excluded from the selector lists).
- `<name>.clash.yaml` ŌĆö one proxy per **Clash-Meta-supported** enabled protocol (VLESS Vision/gRPC,
  Hysteria2, TUIC, Shadowsocks-2022, Trojan) plus a `url-test` proxy-group (`mycelium-auto`) and a
  `select` proxy-group (`mycelium`). ShadowTLS and XHTTP are intentionally **not** represented
  (Clash-Meta lacks support); use the sing-box config for those. Merge the `proxies` and
  `proxy-groups` blocks into your Clash-Meta config.

#### `--engine xray` (optional alternative)

- `<name>.singbox.json` ŌĆö a single sing-box outbound of `type: "vless"` with `flow:
  xtls-rprx-vision`, `tls.enabled`, `tls.server_name` = donor SNI, `tls.utls{enabled,
  fingerprint: "chrome"}`, and `tls.reality{enabled, public_key, short_id}`.
- `<name>.clash.yaml` ŌĆö a single Clash-Meta proxy entry (merge it under your `proxies:` key).

## Params file

Params describe the deploy-time values. **JSON is always supported**; YAML works only if `yq` is
installed (it is converted to JSON internally). See [`params.example.json`](params.example.json),
whose fields are:

**Common fields (both engines):**

| Field | Meaning |
|---|---|
| `engine` | informational; the active engine is chosen by the `--engine` flag |
| `node_address` | public address (host/IP) clients dial; used in subscriptions |
| `donor_host` | the real donor host the REALITY handshake/`dest` points at |
| `donor_sni` | the SNI presented in the REALITY TLS handshake (the donor's name) |
| `reality_private_key` | REALITY private key (server side) ŌĆö from `reality-keys` |
| `reality_public_key` | REALITY public key (client side) ŌĆö from `reality-keys` |
| `short_ids` | array of REALITY shortIds ŌĆö from `reality-keys` |

**xray-engine fields:**

| Field | Meaning |
|---|---|
| `listen_port` | inbound listen port (typically `443`) |
| `dest` | optional explicit REALITY `dest` (defaults to `donor_host:443`) |

**sing-box-engine fields (multi-protocol):**

| Field | Meaning |
|---|---|
| `<proto>_enabled` | per-protocol toggle (`true`/`false`); decides which inbounds/outbounds are emitted |
| `<proto>_port` | per-protocol listen port (`vless_reality_vision_port`, `ŌĆ”_grpc_port`, `ŌĆ”_xhttp_port`, `hysteria2_port`, `tuic_port`, `shadowsocks_port`, `shadowtls_port`, `trojan_port`) |
| `tls_sni` | server certificate SNI for the TLS-cert protocols (Hysteria2 / TUIC / Trojan) |
| `tls_certificate_path` / `tls_key_path` | on-node paths to the server TLS cert / key for those protocols |
| `grpc_service_name` | gRPC service name (VLESS+REALITY+gRPC) |
| `xhttp_path` | XHTTP path (VLESS+REALITY+XHTTP) |
| `shadowtls_handshake_server` / `shadowtls_handshake_port` | upstream the ShadowTLS inbound mirrors (defaults to `donor_host:443`) |
| `ss_password` | Shadowsocks-2022 PSK (base64) ŌĆö also the inner SS of ShadowTLS ŌĆö from `openssl rand`/`sing-box generate rand` |
| `trojan_password` | Trojan password |
| `hysteria2_password` | Hysteria2 password |
| `shadowtls_password` | ShadowTLS handshake password |

The protocol tokens are: `vless_reality_vision`, `vless_reality_grpc`, `vless_reality_xhttp`,
`hysteria2`, `tuic`, `shadowsocks`, `shadowtls`, `trojan`. The legacy/easily-fingerprinted
protocols (VMess, plain pre-2022 Shadowsocks, plain WireGuard, OpenVPN, L2TP/IPsec, PPTP, SSTP,
IKEv2) are **excluded by design** and have no params. AmneziaWG is a **separate** non-TLS/UDP path
(see `nodes/dataplane/amneziawg/`) and is not rendered by this engine.

`params.example.json` is pure JSON and therefore carries **no comment header** (it must stay valid
for `jq`, `sing-box`, and `xray`); its license is documented here, per the repository convention.
Copy it to a gitignored path (e.g. `params.local.json`) and fill in real values; do not edit the
example in place. All `REPLACE_WITH_*` / sentinel values are non-functional placeholders.

## Deploy flow (operator)

The flow below targets the **primary** sing-box engine, which is now the **default** ŌĆö you can omit
`--engine` entirely (the commands below keep `--engine singbox` for clarity). For the optional
single-transport path pass `--engine xray`.

1. **Generate REALITY material** on a trusted machine:
   `myceliumctl reality-keys --shortids 3` ŌåÆ record the private/public keys and shortIds into a
   gitignored params file (e.g. `params.local.json`), filling in `node_address`, `donor_host`,
   `donor_sni`. Generate the protocol passwords (`ss_password`, `trojan_password`,
   `hysteria2_password`, `shadowtls_password`) with `openssl rand`/`sing-box generate rand` and set
   the `<proto>_enabled` flags for the subset of protocols you want to expose.
2. **Issue identities** for the operator(s):
   `myceliumctl identity add --name operator`.
3. **Render the server config**:
   `myceliumctl render-server --engine singbox --params params.local.json` ŌåÆ `state/server.json`.
   Ship that to the node and feed it to `sing-box` (the infra/Ansible component handles placement
   and restart). For the optional engine use `--engine xray` and feed the result to `xray`.
4. **Emit subscriptions** for off-the-shelf clients:
   `myceliumctl subscription --engine singbox --params params.local.json` ŌåÆ `out/`. Distribute the
   per-client configs out of band (outside any infrastructure reachable from heavily restricted
   networks).
5. **Revoke** without redeploying the node:
   `myceliumctl identity revoke <name|id>`, re-render `server.json`, reload the engine.

The runtime **control-agent** (network-state detection, **auto-rotation** of
transport/port/SNI/keys, telemetry) is intentionally **not** part of this tool ŌĆö it is a **Phase-2
deliverable in Go or Rust** (see [`docs/ARCHITECTURE.md`](../docs/ARCHITECTURE.md) Layer 2).
`myceliumctl` covers only the manual, operator-driven render/issue/revoke surface.

## What is gitignored

Everything containing secrets or generated artifacts:

- `control/state/` ŌĆö identity state and the rendered `server.json`
- `control/out/` ŌĆö per-client subscription configs
- `server.json`, `*.local`, `params.local.json`, `secrets/`, `*.key`, `*.pem`

Only placeholders, `*.example`, and `*.template` files are committed. Real keys, UUIDs, IPs, and
domains are generated or provided at deploy time and never land in the repository.

## Tests

```
bash control/selftest.sh
```

The self-test runs with only `bash` + `jq` present. It stubs `xray uuid` / `xray x25519` and
`openssl rand -hex` by prepending a stub directory to `PATH`, then exercises:

- the identity roundtrip (add / list / revoke by name and id);
- **xray engine** ŌĆö `render-server` against the template fixture and `subscription` output;
- **sing-box engine** ŌĆö multi-protocol `render-server --engine singbox` (asserting per-protocol
  toggling prunes a disabled protocol, REALITY/transport/secret sentinels are filled by `jq` path,
  per-protocol user shapes, and no sentinel survives) and `subscription --engine singbox`
  (one outbound per enabled protocol, the `urltest`/`selector` outbounds and Clash `url-test`/
  `select` groups in priority order, ShadowTLS detour wiring, and no private-key leak);
- **engine default + compatibility** ŌĆö the default engine is now `singbox` (no `--engine` renders a
  sing-box config), explicit `--engine xray` still renders the legacy Xray config and subscription,
  and an unknown `--engine` is rejected.

Every generated artifact is verified with `jq`. It exits `0` on success.
